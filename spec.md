# XFIND 仕様書

## 1. 概要

XFIND は、事前に生成したテキストインデックスを用いて高速にファイルやディレクトリを検索する CLI/TUI ツールである。

本プロジェクトは次の 2 コマンドから構成される。

- `xfidx`: 指定ルートを再帰走査し、テキスト形式インデックスを生成する
- `xfind`: 既存インデックスを検索し、結果に対して `open` または `cd` を行う

対象プラットフォームは Ubuntu と X68000 / Human68k とする。
初期実装では、共有コアを重視したクロスプラットフォーム設計を採用する。

## 2. スコープ

### 2.1 初期実装に含めるもの

- 再帰走査によるインデックス生成
- テキスト形式インデックスの保存と読込
- ファイル名およびフルパスの検索
- 大文字小文字を区別しない検索
- TUI による候補一覧表示
- `open` アクション
- `cd` アクション
- Ubuntu と X68000 の両対応を前提とした設計

### 2.2 初期実装に含めないもの

- バイナリインデックス
- 内容全文検索
- 正規表現検索
- 学習ランキング
- 複数選択処理
- 常駐監視
- バックグラウンド自動再索引
- GUI
- 実機 X68000 でのテスト

## 3. コマンド仕様

### 3.1 `xfidx`

指定された 1 個以上のルートディレクトリを再帰走査し、インデックスファイルを生成する。

基本形:

```text
xfidx ROOT [ROOT ...]
```

例:

```text
xfidx /usr/bin
xfidx /home/user/src /home/user/doc
XFIDX.X C:\BIN
XFIDX.X C:\BIN C:\DOC
```

最低限サポートするオプション:

- 出力先指定
- 設定ファイル指定
- 進捗表示抑制

走査中に読めないディレクトリやファイルがあっても、可能な限り継続する。

### 3.2 `xfind`

既存インデックスを読み込み、検索語で候補を絞り込み、選択結果に対して `open` または `cd` を行う。

基本形:

```text
xfind [QUERY]
```

例:

```text
xfind readme
xfind sample
XFIND.X readme
XFIND.X sample
```

最低限サポートするオプション:

- インデックスファイル指定
- 設定ファイル指定
- 非対話モードでの `open`
- 非対話モードでの `cd`
- オープナコマンド指定（設定ファイルの `open_cmd` を上書き）

## 4. インデックス仕様

初期実装ではテキスト形式のみを扱う。

要件:

- 人間が読めること
- デバッグしやすいこと
- 安定した順序で出力されること
- 最低限のヘッダを持つこと

形式例:

```text
XFIND-INDEX 1
ROOT /usr/bin
ROOT /home/user/doc
ENTRY F /usr/bin/vi
ENTRY F /usr/bin/less
ENTRY D /home/user/doc
ENTRY F /home/user/doc/readme.txt
```

各エントリの最小情報:

- 種別 `F` または `D`
- 正規化済みフルパス

必要に応じてパスのエスケープ規則を導入してよいが、初期実装では単純さを優先する。

## 5. 検索仕様

検索対象:

- ベース名
- フルパス

一致方式:

- ベース名の完全一致
- ベース名の前方一致
- ベース名の部分一致
- フルパスの部分一致

比較は既定で大文字小文字を区別しない。

ランキング優先順:

1. ベース名の完全一致
2. ベース名の前方一致
3. ベース名の部分一致
4. フルパスの部分一致
5. ベース名が短いもの
6. フルパスが短いもの
7. 最後は辞書順

結果順は安定していること。

## 6. アクション仕様

### 6.1 `open`

既定アクションは `open` とする。

期待する動作:

- 実行可能ファイルなら直接起動する
- 非実行ファイルならオープナ経由で開く
- ディレクトリに対しては、実装上自然なら `cd` 相当の扱いとしてよい

個別の `edit` や `view` は設けない。
拡張子ごとの処理が必要でも、すべて `open` の内部動作として扱う。

#### オープナの決定順序

非実行ファイルを開く際のオープナは、次の順で決定する。

1. 設定ファイルの `open_cmd` キー
2. 環境変数 `XFIND_OPEN`
3. 自動検出: `xdg-open`、`open`、`mimeopen` の順で最初に見つかったものを使う
4. フォールバック: パスを標準出力に表示し、終了コード 0 で戻る

`xdg-open` は Ubuntu の標準インストールには含まれない場合がある。
3 の自動検出は実行時に `which` 相当の探索を行い、見つからなければ次候補に進む。

設定ファイルの記述例:

```text
open_cmd = xdg-open
```

環境変数の設定例:

```sh
export XFIND_OPEN=xdg-open
```

`open_cmd` および `XFIND_OPEN` には、パスの直前に展開されるコマンドプレフィクスを指定する。
実際の呼び出しは `<open_cmd> <path>` の形式になる。

### 6.2 `cd`

選択したディレクトリ、または選択したファイルの親ディレクトリへ移動する。

親シェルのカレントディレクトリを直接変更できない場合は、各 OS で実用的な補助方式を採用してよい。

## 7. TUI 仕様

最低限必要な操作:

- 候補一覧表示
- カーソル移動
- Enter で `open`
- `c` で `cd`
- `q` で終了

見た目の豪華さより、単純さと保守性を優先する。

## 8. ソース構成

ソースコードは `src/` 以下に配置する。
X68000 依存部は `PLATFORM.H` に集約する。

作成対象ファイル:

- `MXFIDX.C`
- `MXFIND.C`
- `FSWALK.C`
- `FSWALK.H`
- `INDEX.C`
- `INDEX.H`
- `MATCH.C`
- `MATCH.H`
- `ACTIONS.C`
- `ACTIONS.H`
- `CONFIG.C`
- `CONFIG.H`
- `UITUI.C`
- `UITUI.H`
- `PLATFORM.H`

## 9. クロスプラットフォーム方針

- 検索、索引、TUI のコアは可能な限り共通化する
- OS 依存差分は `PLATFORM.H` を入口として分離する
- Ubuntu 上での開発しやすさを重視する
- ただし Ubuntu 専用実装に寄せず、最初から X68000 を意識した保守的な実装にする

## 10. ビルドと配置

- Ubuntu 版はルートの `CMakeLists.txt` でビルドする
- X68000 版は `x68k/Makefile` でビルドする
- GitHub Actions により自動ビルドできる構成とする
